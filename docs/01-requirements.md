# 階段一：需求分析

## 1. 專案定位

ThreadRadar 是一套**商業化的 Threads 關鍵字巡檢與熱門文章監控系統**。核心價值：使用者設定關鍵字與熱門度門檻，系統自動巡檢並在文章首次達標時即時通知，同時提供歷史查詢與趨勢儀表板。

目標特性：高可維護、高擴充、高效能、高可靠性；模組化設計以因應「資料來源策略」未來可能更換（官方 API ↔ 爬蟲 ↔ 混合）。

---

## 2. 功能需求（Functional Requirements）

### FR-1 關鍵字管理
- CRUD 多組關鍵字（名稱、啟用狀態、巡檢頻率、搜尋時間範圍、熱門度條件）
- 啟用/停用開關獨立於刪除（軟開關，保留歷史設定）

### FR-2 搜尋條件
- 時間範圍：30分鐘 / 1小時 / 6小時 / 24小時 / 7天 / 自訂區間
- 熱門度門檻：Views、Likes、Replies、Reposts、Quotes 五個指標
- 每個指標可獨立設定比較運算子：`>` `>=` `=` `<` `<=`
- 一組關鍵字可同時掛多個門檻條件（AND 邏輯，例如 Views>=10000 AND Likes>=500）

### FR-3 文章資料模型
必要欄位：Threads URL（去重鍵）、作者名稱、作者帳號、發文時間、文章內容、圖片、影片、Views/Likes/Replies/Reposts/Quotes、是否驗證帳號、搜尋關鍵字（多對多，同一文章可能命中多個關鍵字）、首次發現時間、最後更新時間。

**Upsert 語意**：以 Threads URL 為唯一鍵。已存在 → 僅更新互動數與 `last_seen_at`；不存在 → 建立新紀錄並記錄 `first_seen_at`。

### FR-4 結果查詢
- 篩選：關鍵字、作者、日期區間、熱門度、是否驗證帳號
- 排序：最新、最熱門（綜合分數）、Views、Likes、Replies、Reposts

### FR-5 通知
- 管道：Email、Discord Webhook、Slack Webhook、LINE（Notify 或 Messaging API）、Telegram Bot、通用 Webhook
- **去重策略**：每篇文章「首次達標」只通知一次（已與使用者確認）。需要 `notified_at` / 達標標記欄位防止重複通知同一篇文章同一條件。
- 通知內容：關鍵字、作者、文章摘要、各項互動數、Threads 連結

### FR-6 自動巡檢
- 頻率選項：1 / 5 / 10 / 30 / 60 分鐘，per-keyword 可獨立設定
- 需要 Scheduler + Queue 架構避免巡檢工作互相阻塞

### FR-7 Dashboard
- 今日統計：搜尋次數、新文章數、更新文章數、通知次數
- 排行榜：熱門文章 / 熱門作者 / 熱門關鍵字（TOP20）
- 趨勢圖：每日文章數、每日通知數、每日熱門文章成長

### FR-8 AI 功能（可選 / 後期迭代）
文章摘要、廣告判斷、情緒分析、主題分類、自動標籤、相似文章合併、每日熱門摘要。**明確標記為 Phase 2+，不影響核心架構的技術選型，但資料模型需預留擴充欄位（如 `ai_summary`, `ai_tags`, `ai_sentiment`）。**

### FR-9 API
REST API，供前端與未來第三方串接：關鍵字 CRUD、文章查詢、Dashboard 統計等端點。

## 3. 非功能需求（Non-Functional Requirements）

| 類別 | 要求 | 設計意涵 |
|---|---|---|
| 高效能 | 巡檢與通知延遲低，Dashboard 查詢秒級回應 | 需要索引設計、快取層、非同步處理 |
| 可水平擴充 | Worker 可依巡檢量水平擴展 | Queue-based worker 架構，狀態不綁死單機 |
| 模組化 | 資料來源（API/爬蟲）可替換不影響其他層 | Provider/Adapter 介面隔離（見 §5） |
| 易維護 | 新增關鍵字/通知管道不需改核心邏輯 | Strategy pattern、設定驅動 |
| 高測試覆蓋率 | 核心業務邏輯（去重、門檻判斷、通知觸發）需單元測試 | 依賴注入、避免與外部 API 強耦合 |
| Docker 支援 | 一鍵啟動開發/生產環境 | docker-compose 涵蓋 app/db/redis/queue worker |
| CI/CD | 自動化測試、build、部署 | 見部署階段文件 |
| 可部署雲端或本地 | 不綁定特定雲廠商 | 避免使用雲廠商專屬託管服務作為硬依賴 |

## 4. 關鍵限制與風險（提前聲明）

本節整理會直接影響技術選型與架構設計的外部限制，詳細研究見 [02-threads-data-source-feasibility.md](./02-threads-data-source-feasibility.md)。

1. **Threads 官方確實有 Keyword Search API**，但需通過 Meta App Review，且有 **每使用者/日 2,200 次查詢** 的硬配額，無公開定價保證（目前免費但可能變動）。
2. **官方 API 配額若不足以支撐商業規模的高頻巡檢**（例如多組關鍵字 × 1 分鐘頻率），會超出 2,200 次/日的上限，需設計「配額感知的排程降級」機制，或混合第三方/爬耲來源。
3. 非官方爬蟲路徑在法律上比預期寬鬆（Meta v. Bright Data 案：登出狀態爬公開資料不受 ToS 約束），但技術對抗成本高（Cloudflare、需登入才能看搜尋結果，除 EEA 地區）。
4. **架構必須做資料來源介面隔離**，讓官方 API / 第三方 API / 爬蟲可以並存或切換，不能寫死單一實作。這是本階段最重要的架構決策，將在系統設計階段（03）具體落實為 `SearchProvider` 介面。

---

## 5. 範圍界定（Out of Scope / 明確排除）

- 不做多租戶（multi-tenant）架構，除非後續使用者提出
- 不做即時串流（WebSocket 即時推播）儀表板，Dashboard 為輪詢/請求式更新
- AI 功能列為 Phase 2，第一版核心系統不依賴任何 AI 服務即可運作

---

## 6. 驗收標準（Definition of Done，第一版 MVP）

- [x] 可建立/管理關鍵字與門檻條件
- [x] 排程可依設定頻率觸發巡檢（資料來源可先接 Mock Provider）
- [x] 文章正確去重（同 URL 更新而非新增）
- [x] 達標文章觸發至少一種通知管道，且同一文章不重複通知
- [x] API 可查詢文章列表（含篩選/排序）與 Dashboard 統計
- [x] 有基本測試覆蓋核心業務邏輯（去重、門檻判斷、通知觸發）
- [x] Docker Compose 可一鍵啟動完整開發環境
